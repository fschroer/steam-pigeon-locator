### #30 (PPS clock) FIXED — uncommitted, host-verified only
The diagnosis held: in `main.c` `HAL_GPIO_EXTI_Callback`, `elapsed` was assigned *before* its own `950000..1050000` sanity check, so two missed PPS edges poisoned it to ~3× and the monotonic clock ran at 1/3 for one interval, leaving every later timestamp ~670 ms early.

Fixed as **more than the one-liner**, because multi-second GPS dropouts are expected under canopy:
- **New pure module** `Rocket/Common/{Inc/PpsDiscipline.h,Src/PpsDiscipline.c}` — `Pps_AcceptInterval()` divides a multi-second delta by the nearest whole second, band-checks the *resulting rate*, and reports the second count. Rejection leaves its output untouched, so the caller holds its last good rate. Split out of `main.c` specifically so it is host-testable — `main.c` can't compile off-target, so an inline test would have been testing a copy.
- **Recovery cap `PPS_MAX_MISSED_SECONDS = 9`**, not the 5 the issue suggested. Derived from `n × 50000 < 500000` given the ±5% band; at n ≥ 10 rounding can yield a plausible-but-wrong rate, which is worse than rejecting. **To cover longer dropouts, tighten the band — don't raise the cap** (±2% → n ≤ 24, ±1% → n ≤ 49). The two constants are coupled; see the ADR-0007 2026-07-18 amendment.
- **Counters** `Pps_GetMissedEdgeCount()` / `Pps_GetRejectedIntervalCount()` distinguish "occluded but disciplined" from "free-running on a held rate". **Nothing reads them yet** → **[#31](https://github.com/fschroer/steam-pigeon-locator/issues/31)** (archive/downlink plumbing; carries an `ARCHIVE_VERSION` question).
- **`Tests/FlightReplay` Part D** (D1–D5) drives the real `Pps_AcceptInterval()`; suite now **97/97** (was 44/44). Mutation-tested — but note Part D guards against a *wrong-but-accepted* rate rather than re-catching the original adopt-on-reject bug, which the refactor made structurally impossible. The ISR wiring in `main.c` is compile-checked only, covered by no host test.

### B. Real-time performance — the big picture (a 6-hour rabbit hole; read this before optimizing anything)
Original `process_dur_us` looked alarming (~25–47 ms). The causes, in order of impact:
1. **This core has NO FPU.** STM32WL5MOC M4 → `-mfloat-abi=soft`, all float software-emulated (IDE offers only FPU=None). **This is hardware, not a bug — do NOT try hard-float.** It means float *op count* is the cost.
2. **Build config.** Debug = `-O0`, Release = `-O3`. The Release config was a stale stub **missing the `Rocket` source folder, the `../Rocket/*` + `Accelerometer`/`BMP280` include paths, and the C++/C language standards (gnu++17/gnu18)** — all present in Debug. Fixed in the IDE (un-exclude `Rocket`, copy includes + std). **Recommended: clone the Debug config and just set -O3** to avoid this whack-a-mole. Fly/measure on -O3.
3. **ITM trace was blocking.** `ITM_Trace::send` spun `while (ITM->PORT==0)` — blocks forever when SWO isn't drained (standalone, or a debugger without the SWO console open). Called 3×/cycle in `MS5611OCCallback` (an ISR) → stalled the ISR → inflated **every** profiled segment, and `-O3` can't speed a busy-wait. **Fixed to non-blocking (drop if port busy).** This was also a real **flight-safety bug** (ITM is enabled in Release; a debugger-less flight would stall). Keep it.
4. **EKF covariance = ~13 ms of soft-float.** Optimized to ~7.6 ms — see **[ADR-0013](adr/0013-realtime-ekf-fpuless-covariance-heuristics.md)**. **EKF kept LIVE for observation** (amends ADR-0005); `m_gps_only_` retired so it fuses through descent.
5. **SPI perf fix #1** (`SpiDevice::readAfterWriteByte`): one `HAL_SPI_TransmitReceive` instead of `Transmit`+`Receive` — removes a HAL call + BSY gap per word; helps all IMU/baro reads in both builds.

**Result:** worst-case `ProcTotal` **47 → 23 ms**, EKF live, comfortably within NFR-3.

### E. Air-start tilt — what's actually wrong (don't relitigate as "gyro drift")
Flight tilt grew monotonically 4.5→123°. Key insight (issue #23): the ~250 dps axial spin **averages out** transverse gyro errors (bias/cross-axis/misalignment → bounded coning over ~1.4 s/rev), so the growth is **largely REAL rotation** (a slow low-power rocket genuinely arcs over + tumbles) plus **unbounded** open-loop drift — NOT the ~6° that simple bias would give. Bare gyro integration is unbounded + unvalidated → unusable for a safety-critical firing gate regardless of accuracy. Fielded air-start systems have a **second attitude reference**; here that's **GPS velocity-vector aiding** (nose ≈ velocity for a stable rocket). The cardinal mounting map is CORRECT (verified by integrating the archived gyro — it reproduces the recorded tilt); the gap is a bounding reference + fine calibration, not the axis mapping.

### F. Heuristics to NOT accidentally break (future context)
- **EKF covariance ([ADR-0013](adr/0013-realtime-ekf-fpuless-covariance-heuristics.md)):** `predict()` computes only the **upper triangle** of `P` (symmetric) and **copies rows/cols 9..14** (bias states = pure-identity `Φ` rows, `kDyn=9`). **If anyone gives the biases dynamics (Gauss-Markov bias, etc.), `Φ` rows 9..14 stop being identity and the copies silently produce a WRONG covariance** — revert to full loops / adjust `kDyn`. `symmetrizeP()` must stay (diagonal floor + used by the non-symmetric measurement updates). Verified numerically identical (host, ~7e-8).
- **No FPU:** optimize float *op count* (structure, not micro-ops); covariance is O(n³) in software.
- **ITM `send` must stay non-blocking.**

## App BLE connection health — probe the receiver instead of tearing down on silence — COMMITTED & BENCH-TESTED (2026-07-05)
Fixed the user-reported "when the app isn't receiving messages from a locator, it repeatedly disconnects and reconnects the receiver" loop. Design & invariants: **[ADR-0012](adr/0012-app-ble-connection-health-probe.md)**; new **NFR-11** + extended **FR-R2**.
- **Root cause:** `BluetoothConnectionManager`'s health watchdog was a one-shot **tear-down-on-silence** timer — 10 s after `Ready` with no GATT data it declared the link a "phantom" and forced a reconnect, which re-armed the watchdog → endless loop. But a healthy receiver relays nothing whenever the locator is quiet (off / on the pad / out of range), so GATT silence ≠ dead link.
- **Fix (app `67f46ab`, `main`):** the watchdog now loops; each 10 s silent window sends a `ReceiverInfoRequest` via a new `onHealthProbe` callback (wired in `BluetoothService` to `requestReceiverInfo()`) — a message the **receiver answers on its own behalf**, independent of the locator. Any inbound byte (relay or probe response) resets the miss counter; the link is reconnected only after `MAX_MISSED_HEALTH_PROBES` (3) consecutive unanswered probes (~30 s). When the locator *is* transmitting, no probe is sent (windows are never silent).
- **Depends on** the receiver answering receiver-info requests unconditionally (FR-R2 / NFR-11). Do **not** reintroduce tear-down-on-silence, and do **not** gate receiver-info replies behind locator activity.
- **Counterpart to** the locator radio-layer CRC change (**[#16](https://github.com/fschroer/steam-pigeon-locator/issues/16)**) — the "paired app-side work" the earlier handoff referenced.
- **Build:** app `compileDebugKotlin` links clean (Android Studio JBR). **Status:** committed + pushed (app `main`). **Bench-confirmed by the user:** app⇄receiver traffic ~every 20 s (probe + reply), receiver stays connected while the locator is idle. No wire-format change.

## Flight-profile transfer reliability — COMMITTED & BENCH-TESTED (happy path) (2026-07-04)
Cross-repo rework of the archived-flight OTA download (locator + receiver + app); design & invariants in **[ADR-0009](adr/0009-flight-data-transfer-reliability.md)**. Fixed a cluster of bench-found defects:
- **App displayed nothing** — it framed `FlightData` by consuming the whole BLE buffer, so bursted/fragmented packets each failed the per-frame CRC-16 and were never ACKed (transfer stalled, locator retransmitted forever). Now delimited by **exact header-derived length**; parity framed at fixed size. (app `d81c22e`)
- **Empty record hung the chart** — locator emits a **`packet_count == 0` "no data" marker**; app shows "No flight data". (locator `d2c1808` + app `d81c22e`)
- **Abort didn't stop the load / stuck locator** — `Process()` aborts the burst on leaving `DataRequested`; app sends `FlightMetadataRequest` on back-out; restored the short idle/active timeouts as a **disconnect safety net** (removing them had let a single lost `DisarmRequest` brick the locator until power-cycle). Resume is app-driven; viewing/empty states use the long 5-min timeout. (locator `d2c1808`)
- **~24 s first-tap delay** — the **receiver** only forwarded app→locator commands in the post-PreLaunchData window, which never opens while the locator is quiet in flight-profile mode; it now forwards immediately once it sees a `FlightMetadata`/`FlightData` frame (flag renamed `in_flight_data_mode_` → `locator_in_profile_mode_`). Locator-internal request→packet-0 latency measured at **178 ms** via temporary UART timestamps (since stripped). (receiver `8f416d2`)
- **Throughput:** burst window `kWindowSize` **4→8**, paired with `kRetxTimeoutMs` 4000→7000; parity group stays 4. (locator `2446fa5`)
- **Diagnostics (kept):** app logs per-transfer health — `missing`/`recovered-by-parity`/`duplicate` counts + a completion summary (logcat tag `FlightDataRepository`). Useful for the #18 loss-validation. (app `f8e338c`)

**Status:** all three repos committed + pushed (locator/receiver `master`, app `main`). Bench-confirmed on the happy path (load, abort, empty record, return-to-main, delay gone). **Loss-condition / at-range validation is open as [#18](https://github.com/fschroer/steam-pigeon-locator/issues/18)**, which also interacts with #16 (radio CRC now admits mismatched frames → the per-frame CRC-16 + header-exact framing is the sole gate).

## Firmware state (on master, mostly UNTESTED on hardware)
- **Strapdown** (`AttitudeEstimator`): seed `quatFromAccel(−accel)` + at-rest gyro-bias learning + negated tilt-correction blend. Drives the orientation display via `getStrapdownQuat` (Y-reflected).
- **NFR-9 high-rate gyro path — IMPLEMENTED & BENCH-CONFIRMED at ~480 Hz (GPS/TIM2):** gyro batched into the ISM6HG256X FIFO at 480 Hz; `ISM6HG256X::drainFifoGyro()` reads the FIFO and `Navigation::Update()` integrates each word, decoupled from the slow path (no 480 Hz SPI ISR — respects NFR-4/NFR-5). **Additive & non-breaking:** if the FIFO is unconfigured/empty (or NAV_TEST replay), it falls back to the single output-register propagate at the loop dt.
  - **Loop rate:** `Update()` runs at the **~20 Hz** super-loop cadence (per the TIM17 tick + once-per-second radio TX; consistent with dt=1/480 ⇒ ~24 words/drain at 20 Hz). An earlier "~150 Hz" claim was a `HAL_GetTick` artifact (see below) — disregard it.
  - **dt is GPS-disciplined, NOT 1/480.** Per-sample dt = (TIM2 interval since last drain ÷ `Pps_GetTim2TicksPerSec()`) ÷ words drained, clamped to [1/4000, 1/120]; holds 1/480 until PPS lock (`m_strapdown_dt_per_word`). This anchors integration to GPS time — immune to MSI inaccuracy, the IMU oscillator (~0.16 %, measured), and **`HAL_GetTick` (see below)**. TIM2+PPS (`elapsed`, exposed via `Pps_GetTim2TicksPerSec` in main.c) is the trusted timebase.
  - **`HAL_GetTick` is RTC-based, NOT SysTick — and is a debugger trap.** `sys_app.c` overrides `HAL_InitTick` (no-op — SysTick never started), `HAL_GetTick` (→ `TIMER_IF_GetTimerValue()`, the RTC/LoRaWAN TimerServer), and `HAL_Delay`. This is the **standard ST LoRaWAN low-power timebase and is correct in production** (~ms resolution; the EKF `dt_s`, sensor timeouts, and `HAL_Delay` across `Rocket/` all rely on it and work). BUT the RTC runs in the backup domain independent of CPU halts, so **under a debugger it freezes or jumps** — which is what produced every phantom rate this session (1477 Hz, 150 Hz loop). **Do not use `HAL_GetTick` to measure anything while halted at a breakpoint; use TIM2.** Not a firmware bug — by design.
  - **Gyro ODR fixed at 480 Hz (was 60 Hz — bug).** `configureOdrAndRanges` tied the gyro `CTRL2` ODR to `sample_rate_hz` (clamped to 20 → `0b0101` = 60 Hz), starving the 480 Hz FIFO of real samples (bench showed n≈7–10/loop, not ~24). Gyro ODR is now a fixed `ODR_G_480` (`0b1000`), decoupled from the 20 Hz loop and matched 1:1 to `FIFO_BDR_GY_480`. This also fixes a latent integration-rate mismatch: `propagate()` uses `dt=1/480`, which was wrong while the gyro truly ran at 60 Hz. The low-g accel (`CTRL1`) intentionally stays at the loop-rate ODR (tilt correction is quasi-static, 20 Hz).
  - **FIFO register block confirmed against ISM6HG256X DS15034 Rev 2** (datasheet at `docs/datasheets/ism6hg256x.pdf`): `FIFO_MODE=110b`=continuous (Tbl 42), gyro `TAG_SENSOR=0x01` (Tbl 232), `DIFF_FIFO_8`=`FIFO_STATUS2` bit 0 (Tbl 81).
  - **BDR_GY must be set ABOVE the gyro ODR (`1001b`, not `1000b`).** Per Tbl 40, `1000b`=480 Hz and `1001b`=960 Hz; per Tbl 57, ODR_G=`1000b`=480 Hz. EMPIRICALLY, BDR == ODR (both 480) batched at **half** rate (GPS/TIM2: 232 Hz); BDR one notch above (`1001b`) captured every sample (~480 Hz, no duplication — ODR caps the stream). **The datasheet documents no BDR-vs-ODR rule; the cause of the BDR==ODR halving is unconfirmed.** (A prior "table offset by one" note was wrong; `1000b` genuinely *is* 480 Hz in Tbl 40.) To nail the mechanism, an isolating test is ODR=240/BDR=480. The earlier `1477 Hz`/`3×` numbers were `HAL_GetTick` artifacts, never real.
  - **Readout is per-word (1 tag + 6 data, one SPI transaction each):** the datasheet documents only a single word across `0x78–0x7E`, so a multi-word burst is *not* relied on (reading past `0x7E` walks into regular registers, not the next FIFO word). ~3 words per `Update` (480 Hz ÷ ~150 Hz) × 7 B is trivial on the bus.
  - **Drained in small batches — do NOT reintroduce a large stack buffer.** `Navigation::Update()` is in the deepest periodic call chain (→ `InsEkf15::updateGps*` ~520–680 B) on a **2 KB stack** (`_Min_Stack_Size = 0x800`). A `Vec3f fifo_gyro[48]` (576 B) local pushed `Update()`'s frame to 992 B and **overflowed the stack** — observed as NaNs in the drained array and an `n` inconsistent with the valid data. Fixed by draining/integrating in 12-word batches (`kStrapdownFifoBatch`, looped to `kStrapdownFifoMaxBatches`), keeping the buffer at 144 B and `Update()` at ~568 B. Verify frame size via `Navigation.su` if this code changes.
  - **FIFO flushed on (re)seed:** `ISM6HG256X::fifoFlush()` (bypass→continuous) is called wherever the strapdown is seeded (`Navigation` pad-rest seed + mounting-cal re-seed), so the first post-seed drain integrates only post-seed samples — never stale pre-seed/pre-launch rotation backlogged in continuous mode. Steady state produces ~3 words per `Update` vs a 12×48 drain ceiling, so no backlog re-forms.
